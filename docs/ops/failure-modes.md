# Режимы отказов и инъекции сбоев

## 1. Цели проверки

Цель — подтвердить идемпотентность и корректные ретраи при частичных сбоях без порчи состояния.

## 2. Сценарии инъекции

- Таймаут/ошибка БД при получении audit‑событий: доставка переводится в retry, без DLQ и без дублирования доставки.
- Недоступность Data Plane при dispatch: статус обновляется в `error`, повторный dispatch не должен создавать дубль.
- Недоступность webhook/SIEM sink: доставка получает retry с детерминированным backoff, попытки append‑only.
- Retry‑storm (серия 5xx): увеличивается `attempt_count`, `next_attempt_at` монотонно растёт, delivery остаётся тем же.

## 3. Критерии корректности

- Уникальность delivery сохраняется (нет дублей по `sink_id + event_id` и `subscription_id + event_id`).
- Повторные попытки не меняют бизнес‑состояние вне предусмотренных таблиц очереди.
- Переход в DLQ происходит только по достижении max‑attempts или перманентной ошибке.

## 4. Проверки в тестах

- Audit export: retry при недоступном событии и сохранение состояния очереди.
- Webhooks: рост `attempt_count` и backoff при повторных 5xx.
